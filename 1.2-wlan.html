1.2 - WLAN

Autonomous design
...

Every access point is like a layer-2 switch, with an IP address defined on a BVI to manage it. 2 main issues with a WLAN deployment using only autonomous APs:

  1. Layer 3 roaming: two distant APs in a campus have probably a layer 3 boundary between them. For a smoother roaming, we would need to span the layer 2 domain over a broader scale. That's bad design.
  2. Management: with dozens of APs, manual config and configuration sprawl are pain points. One solution was WLSM and WLSE, pushing CLI configuration in a more consistent way.

The solution is to use Lightweight Access Points (LAPs)

Lightweight design
...

Using a central controller: WirelessLanController (WLC)

Every LAP will tunnel its traffic to the WLC (Control and Dataplane) using CAPWAP (Control And Provisioning of WirelessAccessPoints).
It solves the issue of L3 roaming, as all endpoints are part of a single L2 domain.
This L2 domain has to be separated from the LAN where the WLC is deployed (usually the DC).
It gives a single pane of glass for visibility and management

WLC deployment
...

  1. Centralized: Standard LAP design. Could be hard to scale to thousand of APs
  2. Distributed: Deploy multiple WLCs, one in each site.
  3. Branch (Flex Connect): The APs only tunnel control plane data to the WLC, the dataplane is sent directly from the AP. More efficient traffic path, but less roaming capabilities
  4. Cloud (Meraki): The WLC is a cloud service

FlexConnect
...

Allows to setup 2 types of WLAN, both use central authentication by default:

  • Centrally switched - the data tunnel goes to the HQ via the WAN link.
  • Locally switched - the data tunnel is terminated locally at the branch.
    A locally switched WLAN can also be configured for Local authentication.

If the WAN link goes down, the AP switches to Standalone mode. All Centrally switched WLANs become unusable. The locally switched WANs keep functioning

  • for existing sessions only if central authentication is configured
  • for existing and new sessions only if local authentication is configured

Coverage

at least 20% overlap, or even 35% for better voice/video experience
We want to stay above -67dBm
We also try to use different channels for each overlapping signal
