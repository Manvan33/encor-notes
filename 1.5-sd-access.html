1.5 - SD-Access

Benefits
...

Similar advantages to SD-Wan, but in the access layer

  • Spanning tree is a failure of Ethernet
    We want to scale L2, to avoid the hassle of L3 roaming
    We remove STP by using a L3 overlay to span a L2 domain
  • SD-Access aims to limit configuration mistakes and inconsistencies.
  • Helps with security and policy
    • By separating location and identity.
      Security rules often use the IP Address to identify devices, but the IP also depends on your location, so it isn't good. By having an identity independent from your location, you can better enforce policies
    • QoS is different on each platform. We need a unified way to configure QoS.

you can enforce policies based on who you are, not where you are

And same benefit than SDN = creating tunnels easily

Components
...

Underlay is composed of physical equipment:

  • Switches, routers, WLC, APs
    Controllers:
  • DNA (Digital Network Architecture) Center -> Only a physical appliance, no VM (for now?)
  • Identity Services Engine (ISE): Track people's identities -> appliance or VM

4 Layers:

  • Physical
  • Network (control plane & data plane)
  • Controller
  • Management

Operation
...

Campus fabric = overlay = mesh of tunnels on top of the underlay = CTS + VXLAN + LISP
The underlay devices can have different roles:

  • Fabric edge nodes - Access switches
  • Control plane nodes - LISP Node containing Mapping database of all the users and their location
  • Fabric border node - Between the campus fabric and other networks. May be Internal or External. The Fabric border nodes can be the control plane nodes (doesn't scale well, better keep them at the distribution layer)
  • Fabric WLC - CAPWAP tunnel with the APs, but the dataplane of the AP still goes to the Fabric Edge node.
  • Everything else is Intermediate nodes

Network layer
...

VXLAN for Data Plane
LISP for Control Plane
Cisco TrustSec (CTS) for Policy Plane

For the underlay, we can use the existing network
L2 or L3 (prefered), and an IGP (IS-IS) and support Jumbo frames.
If the existing network is Greenfield, we can use DNA-Center to deploy everything automatically.

Management layer and controller
...

DNA-Center is the controller, and it only does OOB (out-of-band) management. So if it goes down, the network still works, you just can't change the configuration

It has 2 subsystems:

  • NCP - Network Control Platform (was APIC-EM before). Automates the underlay and overlay
  • NDP - Network Data Platform. Assurance engine: collects data using SNMP, Netflow, Telemetry... To detect errors and suggest resolutions

ISE talks with DNA-Center using REST APIs

DNA-Center pushes the policies into PAN (Policy Administrative Node of ISE) for ISE to enforce them.

The management Layer is achieved by the GUI and REST interfaces of DNA-Center.
Following a 4-step workflow:

  • Design
  • Policy
  • Provision
  • Assurance
