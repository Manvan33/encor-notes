CLI passwords
...

enable <priv> allows to get the <priv> level, or 15 by default
3 types of password protection:

  • enable secret password
  • line password
  • username password

Enable secret
...

Better than enable password
by default, the enable secret is stored as a MD5 hash

Router(config)# enable algorithm-type <algo> secret MySecret

available algorithms, from weaker to stronger:
md5 Encode the password using the MD5 algorithm - type 5
sha256 Encode the password using the PBKDF2 hashing algorithm - type 8
scrypt Encode the password using the SCRYPT hashing algorithm - type 9

Line passwords
...

password required when connecting to the console or tty lines

Router(config)# line con 0
Router(config-line)# password cisco
Router(config-line)# login

The password is stored in clear text by default.
service password-encryption enables encryption, but type 7 (vigenère) by default.
We prefer user-based passwords, by using login local

User-based passwords
...

Router(config)# username ivan password cisco

Stores the password in cleartext by default, or type7-encrypted if service password-encryption is enabled.
username ivan secret cisco will store a type-9 hash.
username ivan algorithm-type <algo> secret cisco allows to choose the encryption type.
Specifying a privilege level sets the default level, without the need for enable.

AAA
...

Authentication: identifying a user before allowing access to a protected resource.
Also, the authentication component may offer other services, depending on the security protocol selected, such as, for example, additional challenge and response, messaging support, and encryption.

Authorization: After the user gains access to the network, allows you to control the level of access that users have.
For example, via authorization you can define which privileged EXEC commands are available to the user, or you may control remote access by allowing a user to use protocols such as PPP or Serial Line Internet Protocol (SLIP). User capabilities are defined by a set of attribute-value pairs, which are associated with the user or user group. These pairs may be stored locally on the device or on the centralized TACACS+ or RADIUS server.

When a user tries to perform an action, the AAA engine on the device queries the authorization server for that specific attribute and user. Based on the reply from the server, the user is allowed or not allowed to perform that specific function.

Accounting: Accounting is performed after authentication, enables you to collect information about the activity and resource consumption for a user.
keep track of user logins, commands executed by the user, session durations, bytes transferred, and so on. A network device sends this information in a form of attribute-value pairs to the accounting server. This information can then be used for billing, auditing, and reporting purposes.

Authentication can be valid without authorization and accounting. Authorization and accounting, however, cannot be performed without authentication.

Authentication, Authorization, Accountability (AAA) has many benefits:

  • authorization flexibility on a per-command or per-interface level, which is unavailable with local credentials.
  • Scalability, helps track outdated credentials, since the user database is centrally managed instead of across multiple network devices.
  • Standardized and proprietary authentication methods like RADIUS
  • Redundancy

Authentication
...

AAA authentication can be used for different type of services, with a list of methods to be tried consecutively

R1(config)# aaa authentication <service> default|<name> method1 [method2, ...]

Some methods:
enable Use enable password for authentication.
group Use Server-group
krb5 Use Kerberos 5 authentication.
line Use line password for authentication.
local Use local username authentication.
local-case Use case-sensitive local username authentication.
none NO authentication.

Radius
...

Standard protocol defined in RFC2865 and RFC2866, mainly used for network access
Uses UDP/1812 and UDP/1813
Authorization and Authentication are combined
A RADIUS server’s IP address is defined under the radius server configuration mode using address ipv4 192.168.102.101
Only the password is encrypted in packet exchanges

TACACS+
...

Cisco proprietary, mainly used for device admin access
Uses TCP port 49
auth the user once, but still asks for authorization every time they try to enter a command.
The whole packet is encrypted

Configuring AAA
...

  • Enable AAA services in global configuration mode.
  • Configure a local username to prevent loosing access
  • Configure AAA via radius + fallback on local
SW(config)# aaa new-model
SW(config)# username admin secret adminpass
SW(config)# aaa authentication login default group radius local

The aaa new-model command immediately applies local authentication to all lines and interfaces (except the console line line con 0)

Configuring RADIUS server
...

Configure the RADIUS server

R1(config)# radius server RADSRV
R1(config-radius-server)# address ipv4 10.255.255.101 auth-port 1812 acct-port 1813
R1(config-radius-server)# key SecretRAD
  • Associate the RADIUS server with a server group.
R1(config)# aaa group server radius RADSRVGROUP
R1(config-sg-radius)# server name RADSRV
  • Configure aaa authentication login to use a server group with a fallback to local authentication.
R1(config)# aaa authentication login SRVAUTH group RADSRVGROUP local

To apply this auth method to vty lines:

R1(config)# line vty 0 4
R1(config-line)# login authentication SRVAUTH

TACACS and secure keys storage
...

When configuring a TACACS+ server, you will probably provide the shared key used to encrypt authentication packets.
To store this key in a secure way, you need to enable key encryption on the router, and specify a master key. More info

R1(config)# key config-key password-encrypt <master-key>
R1(config)# password encryption aes

You can then provide the shared key in the TACACs server config:

R1(config)# tacacs server TACSRV
R1(config-server-tacacs)# address ipv4 <IP@> 
R1(config-server-tacacs)# key <shared-key>

To test authentication against the configured TACACS server:

R1# test aaa group tacacs+ <user> <password> new-code

TACACS+ authentication flow is the following:

  1. --> START
  2. <-- REPLY to ask for the username
  3. --> CONTINUE with the username
  4. <-- REPLY to ask for the password
  5. --> CONTINUE with the PASSWORD
  6. <-- REPLY containing the result of the authentication

Configure authorization and accounting
...

R1(config)# aaa authorization exec MYTACAUTH group TACSRVGROUP local if-authenticated
R1(config)# aaa authorization commands 15 MYTACAUTH group TACSRVGROUP local
R1(config)# aaa authorization config-commands

Apply this method to vty lines

Router(config)# line vty 0 4
Router(config-line)# authorization exec MYTACAUTH
Router(config-line)# authorization commands 15 MYTACAUTH
